# 远程勘查板块重构设计

## 背景

当前远程勘查板块存在以下问题：

- `公网远勘` 与 `CTF / Lab` 语义混杂，用户心智不清晰。
- 页面元素过多，统计卡、说明块、操作按钮堆叠，阅读负担高。
- `CTF / Lab` 依赖通用智能体，不符合本次“自建独立远勘智能体”的要求。
- 远勘、内网扫描、服务器取证之间的边界不够清晰。

本次改造仅限远程勘查板块，不影响服务器取证、通用智能体、内网扫描原有能力。

## 目标

- 将远程勘查板块重构为两种模式：`公网远勘` 与 `深度远勘`。
- 将 `CTF / Lab` 改名并重定位为 `深度远勘`，用于专业深度信息获取与案件研判。
- 新建独立的远勘智能体，不复用 `GeneralAgent`。
- 降低页面复杂度，默认先展示结论，细节按需展开。
- 优化公网远勘结果，使每个目标都能产出一段清晰的办案分析。

## 信息架构

### 公网远勘

定位：基础信息探测与线索采集。

输入：

- 目标网址列表
- 侦查备注
- 采集深度

输出：

- 站点用途判断
- 技术架构判断
- 风险方向研判
- 可调证线索
- 同源拓线建议
- 下一步人工核验建议

默认仅展示摘要卡片和按目标折叠的分析卡，原始证据放在二级展开中。

### 深度远勘

定位：专业深度信息获取与结构化研判。

输入：

- 目标网址 / 域名
- 案件说明
- 已知线索
- 补充证据
- 侦查重点

输出：

- 重点发现
- 数据库 / 泄露信息线索
- 嫌疑人关联信息
- 后台与站点控制研判
- 可调证信息
- 后续侦查建议

深度远勘使用独立智能体组件，拥有自己的会话、提示词、状态管理和报告生成逻辑。

## 组件设计

### PentestPanel

保留为远程勘查板块主容器，但只负责：

- 顶部模式切换：`公网远勘`、`深度远勘`、`内网扫描`
- 路由到不同子视图

其中：

- `公网远勘` 使用轻量表单 + 摘要卡 + 目标分析卡
- `深度远勘` 使用独立智能体工作台
- `内网扫描` 保持原能力，但视觉上与远勘工作流分离

### 新增组件

- `ReconAgent.tsx`
  - 独立深度远勘智能体
  - 基于 `runConversationLoop` + `sendToAI`
  - 不依赖 `GeneralAgent`
- 可选拆分：
  - `PublicReconWorkspace.tsx`
  - `DeepReconWorkspace.tsx`

若时间有限，可先将上述逻辑保留在 `PentestPanel.tsx`，后续再拆组件。

## 数据与执行设计

### 公网远勘

继续复用：

- `web_recon_batch`
- `web_recon_progress`
- `buildInvestigationClues`
- `summarizeWebReconTarget`

改造重点：

- 简化默认 UI
- 新增逐目标结论卡
- 重写报告摘要结构，使结果更偏办案研判而不是技术枚举

### 深度远勘智能体

复用底层：

- `sendToAI`
- `runConversationLoop`
- `AIMessage`
- `AISettings`

专属内容：

- 自定义系统提示词
- 自定义 `generalInfo` 上下文拼装
- 自定义工具集合
- 自定义工具执行器
- 自定义消息展示和报告生成

初期可先以“分析型智能体”为主，不强依赖复杂工具链。必要时逐步接入：

- 公网远勘结果作为上下文
- Web 检索/网页抓取类工具
- 已保存证据池

## 报告设计

### 公网远勘报告

报告以“逐目标分析”为核心。

每个目标固定输出：

- 网站用途判断
- 技术架构画像
- 重点风险方向
- 可调证线索
- 同源拓线建议
- 人工核验建议

同时保留全局摘要：

- 目标总数
- 可访问数
- 主要技术栈
- 重点调证线索
- 同源聚类

### 深度远勘报告

报告为案件研判稿，结构偏结论导向：

- 任务概况
- 重点发现
- 数据库 / 泄露信息 / 嫌疑人关联线索
- 后台与权限控制研判
- 可调证信息
- 后续侦查建议

## 错误处理

- 公网远勘失败时仅提示当前目标失败，不淹没整页内容。
- 深度远勘智能体网络、API 或工具失败时，将失败原因以简洁状态条展示。
- 长过程默认不展示冗余 thinking，只展示必要状态。

## 验证方案

- 远勘页面能正常切换 `公网远勘`、`深度远勘`、`内网扫描`
- `深度远勘` 不再出现 `CTF`、`Lab`、`flag`、`题目类型` 等文案
- `深度远勘` 不再写入或打开 `GeneralAgent`
- 公网远勘能正常调用原有 `web_recon_batch`
- 导出功能仍可生成 Markdown / JSON
- 关键改动文件通过 TypeScript / linter 基础诊断
