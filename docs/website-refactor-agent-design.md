# 网站自动重构 Agent 设计方案（服务器可操作版）

## 1. 目标与定位

本 Agent 目标是：在**不丢证据、可回滚、可审计**前提下，自动完成网站重构与安全加固。

适用场景：
- 传统单体站点（PHP/Java/Node/Python）重构为可维护结构
- 老旧配置（Nginx/Apache/Tomcat）治理
- 竞赛/应急场景下的网站快速修复与恢复上线

## 2. 借鉴竞赛思路（FIC / 美亚杯 / 盘古石）

结合公开复盘材料，可抽象出服务器赛题高频解题路径：
- 先定时间线，再定入口，再定落点（日志→上传点→后门/篡改文件）
- 优先证据保全，操作要可回溯（哈希、快照、变更记录）
- 快速区分业务故障与入侵痕迹（配置错误、弱口令、恶意文件、异常进程）
- 通过多源交叉验证（Web日志、系统日志、进程连接、计划任务、自启动项）
- 最后才进入修复与重构，且必须保留回滚路径

## 3. Agent 总体架构

### 3.1 四层架构

1) Orchestrator（编排层）
- 负责阶段推进、风险决策、任务中断与恢复

2) Forensic Guard（取证守卫）
- 快照前置校验、证据归档、变更审计、回滚点管理

3) Refactor Worker（重构执行）
- 代码与配置重构、依赖整理、目录重排、服务编排改造

4) Verifier（验证闭环）
- 功能验证、性能基线对比、安全基线对比、上线验收

### 3.2 可用工具组合

- SSH 命令执行：探测与改造（受策略约束）
- MCP Browser：改造后 UI/交互回归检查
- 本地/远程文件能力：配置备份、差异比对、产物打包
- AI 推理：生成重构计划、风险清单、修复补丁建议

## 4. 强制“快照前置”机制（核心）

Agent 在执行任何写操作前必须先通过快照闸门：

### 4.1 Snapshot Gate 状态机

- `INIT`：收到任务
- `NEED_SNAPSHOT`：提示并要求用户执行快照
- `WAIT_CONFIRM`：等待用户确认“快照完成”
- `EVIDENCE_LOCK`：记录快照编号/时间/目标主机
- `EXECUTION_READY`：允许进入改造阶段

### 4.2 强约束规则

- 未进入 `EXECUTION_READY` 前，禁止执行写操作命令
- 若用户拒绝快照，只允许“只读诊断模式”
- 快照确认信息必须写入审计记录（操作人、时间、目标、备注）

## 5. 自动化重构流程（8 阶段）

1) 资产识别
- 识别语言栈、框架、中间件、部署方式、依赖管理工具

2) 基线采集
- 采集可用性、性能、错误率、安全暴露面

3) 风险扫描
- 识别后门特征、弱配置、高危目录权限、明文密钥

4) 方案生成
- 生成“最小破坏”重构计划（结构、配置、依赖、发布）

5) 变更预演
- 先干跑（dry-run）输出拟修改清单与影响评估

6) 分批执行
- 按模块小步提交：配置→静态资源→后端→数据库迁移（如有）

7) 自动验证
- 健康检查、关键路径接口、页面可访问、日志错误回归

8) 交付与回滚卡
- 输出变更报告、风险余项、回滚步骤、后续优化建议

## 6. 命令与权限治理

### 6.1 命令分级

- L0 只读：`cat` `grep` `find` `ls` `ps` `ss` `journalctl`
- L1 低风险写：备份、复制、权限收敛
- L2 高风险写：删除、覆盖配置、服务重启、迁移

默认仅放行 L0/L1，L2 需要二次确认。

### 6.2 禁止项

- 交互式编辑器和阻塞式 TUI
- 无备份直接覆盖核心配置
- 未快照执行任何破坏性命令
- 未记录变更直接重启关键服务

## 7. 可执行输出规范（Agent 对用户）

每轮固定输出：
- 当前阶段
- 已确认快照信息
- 本轮计划执行命令（按风险等级标注）
- 预期影响
- 实际结果与下一步

最终交付包含：
- 重构前后对比
- 风险与未决项
- 回滚卡（一步一步命令）
- 运维建议（监控、备份、补丁、最小权限）

## 8. 默认重构策略（可直接落地）

1) 结构治理
- 分离 `config/`、`runtime/`、`logs/`、`uploads/`

2) 反向代理治理
- 统一 TLS、超时、限流、错误页、真实 IP 透传

3) 安全治理
- 关闭目录浏览、限制上传可执行、密钥环境变量化

4) 发布治理
- 蓝绿/金丝雀优先；最少支持“单机备份+原地回滚”

5) 可观测性治理
- access/error 日志标准化，关键接口健康探针

## 9. 失败与回滚策略

- 任一关键验证失败，立即进入 `ROLLBACK_PENDING`
- 自动给出回滚命令并等待用户确认
- 回滚后自动复验可用性并输出事故简报

## 10. 你可以直接这样使用

把该 Agent 设为“网站重构专家”，并将首条任务固定为：

1. 先提醒用户做服务器快照并等待确认  
2. 快照确认前仅允许只读探测  
3. 生成重构计划并分批执行  
4. 每批次后自动验收，失败自动进入回滚流程  

## 11. 已接入的竞赛场景技能包

当前软件已内置以下竞赛场景技能包（由系统自动路由）：

- 快照闸门（默认开启）
- 宝塔场景
- 集群场景（Master/Node 分工）
- K8s 场景
- Java 站点
- SpringBoot 场景
- 网站服务中间件场景

当前版本已升级为“自动路由 + 两阶段模式”：
- 阶段1：先只读探测并识别框架；
- 阶段2：识别后自动调用对应skills执行分析/重构；
- 用户侧不显示手动勾选，仅显示“识别到 xxx 框架，调用 xxx skills”状态。

交互形态已调整为“同一对话框模式切换”：
- 在通用智能体对话框内可切换 Sole Coder / Solo Builder；
- 不再通过单独侧边栏入口区分网站重构智能体。
